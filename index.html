ERROR HANDLING AND DEBUGGING IN JAVASCRIPT

JavaScript uses a concept called execution context. They include: Global Context: Any code outside a function. Covers the whole page. Function Context: Any code residing inside a function. Eval Context

Whenever you interdivter comes across an error in your code, it'll throw an exception. It'll check the current execution context for an exception handling code, then it'll move higher up the stack. If it reaches the global context and doesn't find an exception handler, it throws an error object. The error object will help you identify where the error is.

7 Types of built in error objects in JS:

  1. Error: all other errors are based upon this. Is a template from which all other errors are created.
  2. Reference Error: an undeclared variable has been referenced.
  3. Type Error: unexpected data type that cannot be coerced.
  4. Range Error: Numbers not in acceptable range.
  5. URI Error: encodeURI(), decodeURI(), and similar methods not used correctly.
  6. EvalError: eval() function used incorrectly.
  7. isNaN: performing mathematical operations with strings.
Handling errors:

Use try, catch, throw, finally statements. Debugging is about deduction: eliminating potential causes of an error.

DEBUGGING WORKFLOW

Narrowing down to the problematic area

  1. Look at the error message:(browser dev tools): It'll tell you:
    • The relevant script that caused the problem.
    • The line number where it became a problem for the interdivter. (note that this is the point from which script execution could not continue and not necessarily the cause of the error).
    • The type of error (the underlying cause may be different).
  2. Check how far the script is running. Use tools to write messages to the console to tell how far your script has executed.
  3. Use breakpoints where things are going wrong. They let you pause execution and inspect the values that are stored in variables.

    4. If you are stuck on an error, describe the situation to another programmer. Explain what should be happening and where the error appears to be happening. If nobody else is available, try describing it to yourself.

    When you are debugging errors, it can be of help if you look at the error in more than one browser as they can show different error messages.

THE CONSOLE

Browsers that have a console have a console object. Which has several methods your script can use to display data to the console.

CONSOLE METHODS

            
            console.log();
            /*From here you'll notice that these methods have different colors and icons to distinguish them:*/
            console.info();
            console.warn();
            console.error();

GROUPING MESSAGES:

When writing a set of related messages to the console, use console.group(); This allows you to expand and contract the results. It has one parameter, the name that you want to use identifying the grouped messages. 

    
        console.group('Area calculations');
        console.info('Width ', width);
        console.info('Height ', height);
        console.log(area);
        console.groupEnd();

BREAKPOINTS

You can pause the execution of a script on any line using breakpoints. Then you can check the values stored in variables at that point in time.

CHROME

  • Select Sources option
  • Select the script you are working.
  • Find the line number you want to stop on and click on it.
  • When you run the script, it'll stop on this line.
  • You can now hover over any variable to see its value at that time in the script's execution.

FIREFOX

  • Select the Debugger option.
  • Select the script you are working with.
  • Find the line number you want to stop on and click on it.
  • When you run the script, it'll stop on this line, you can now hover over any variable to see its value at that time in the script's execution.

If you set multiple breakpoints, you can step through them one-by-one to see where values change and a problem can occur. When you set breakpoints, the debugger lets you step through the code line by line and see the values of variables as your script progresses. When you are doing this, if the debugger comes across a function, it'll move onto the next line after the function. (It doesn't move to where the function is defined). This behavior is called stepping over. It is possible to tell the debugger to step into a function to see what is happening inside the function.

buttons:

  • pause
  • play
  • step through: go to the next lines one-by-one.
  • step into: a function call. The debugger will move to the first line in that function.
  • step out: of a function that you stepped into.
CONDITIONAL BREAKPOINTS

You can indicate that a breakpoint should be triggered only if a condition that you specify is met. The condition can use existing variables.

CHROME

  • Right-click on a line number
  • Select Add Conditional Breakpoint
  • Enter a condition into the popup box
  • When you run the script, it'll only stop on this line if the condition is true

FIREFOX

  • Right-click on a line of code
  • Select Add Conditional breakpoint
  • Enter a condition into the popup box
  • When you run the script, it'll stop in this line only if the condition is true

DEBUGGER KEYWORD

You can create a breakpoint in your code using just the debugger keyword. When the developer tools are open, this will automatically create a breakpoint. You can also place the debugger keyword within a conditional statement so that it only triggers the breakpoint if the condition is met. It is particularly important to remember to remove these statements before your code goes live as this could stop the page running if a user has developer tools open.

TIP: If you have a development server, your debugging code can be placed in conditional statements that checks whether it is running on a specific server (and the debugging code only runs if it is on the specified server).

HANDLING EXCEPTIONS

    
      try {
        //Try to execute this code
      } catch (exception) {
        //If there is an exception, run this code
      } finally {
        //This always gets executed
      }

If you use a continue, break, or return keyword inside a try, it will go to the finally option.

THROWING ERRORS

If you know something might cause a problem in your script, you can generate your own errors before the interdivter creates them.

throw new Error('message');

This creates a new Error object using the default Error object.
